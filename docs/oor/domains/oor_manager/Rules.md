# Rules - правила и инварианты

Правила проверок для домена oor_manager. Нарушение любого применимого правила блокирует операцию (создание/изменение сущности или выполнение перехода).

---

## Инварианты для мандатов

### R-MANDATE-TRANSITION-EXISTS

**Мандат не может ссылаться на несуществующий переход.**

- **Область:** сущность `Authorization_Mandate`, поле `transition_id`.
- **Условие:** если `transition_id` задан, в домене должна существовать запись перехода (Transition) с таким UUID.
- **Проверка:** при создании и обновлении мандата - запрос к справочнику переходов; при отсутствии перехода - ошибка валидации.

### R-MANDATE-METHOD-EXISTS

**Мандат не может ссылаться на несуществующее определение метода.**

- **Область:** сущность `Authorization_Mandate`, поле `method_definition_id`.
- **Условие:** если `method_definition_id` задан, должна существовать сущность `Method_Definition` с таким UUID.
- **Проверка:** при создании и обновлении мандата - проверка существования метода; при отсутствии - ошибка валидации.

---

## Инварианты для переходов

### R-TRANSITION-FROM-STATUS

**Переход возможен только из допустимого предыдущего состояния.**

- **Область:** выполнение перехода (смена `status` у Domain_Module).
- **Условие:** текущее значение `status` сущности должно совпадать с полем **From** выбранного перехода.
- **Проверка:** перед применением перехода - сравнение текущего статуса с `from_status`; при несовпадении - отказ.

### R-TRANSITION-MANDATE-REQUIRED

**Для выполнения перехода у субъекта должен быть мандат на этот переход.**

- **Область:** выполнение перехода.
- **Условие:** роль текущего пользователя (или контекста) имеет мандат, у которого `transition_id` совпадает с идентификатором выполняемого перехода.
- **Проверка:** перед применением перехода - проверка прав по таблице мандатов; при отсутствии мандата - отказ.

## Инварианты для сущностей

### R-FROZEN-IMMUTABLE

**Сущность в статусе Frozen не может быть изменена.**

- **Область:** обновление Domain_Module (и других артефактов со статусом Frozen).
- **Условие:** если `status === Frozen`, запрещены любые изменения полей (кроме аудита). Для изменений необходимо создать новую версию в статусе Draft.
- **Проверка:** при обновлении сущности - проверка текущего статуса; при Frozen - ошибка (кроме разрешённых служебных полей по политике домена).

### R-DOMAIN-MODULE-PROJECT-EXISTS

**Доменный модуль должен ссылаться на существующий проект.**

- **Область:** сущность `Domain_Module`, поле `project_id`.
- **Условие:** в системе существует `Requirement_Project` с таким `id`.
- **Проверка:** при создании и обновлении Domain_Module - проверка существования проекта.

### R-ENTITY-INHERITS-BASE-DOCUMENT

**Все сущности должны явно указывать наследование от [Base] Document.**

- **Область:** все сущности домена (Requirement_Project, Domain_Module, Method_Definition, Authorization_Mandate).
- **Условие:** в документации сущности должен быть указан факт наследования от [Base] Document и соответствие атрибутов.
- **Проверка:** при генерации документации - проверка наличия раздела "Наследование" или явного указания.

## Правила генерации документации

### R-DOC-GEN-MANDATE-FORMAT

**Мандаты должны генерироваться в стандартном формате таблиц.**

- **Область:** генерация файла Mandates.md.
- **Условие:** мандаты должны быть представлены в таблицах с колонками: ID мандата, Тип, Целевой объект, Условия, Описание.
- **Проверка:** при генерации документации - проверка соответствия формату.

### R-DOC-GEN-TRACEABILITY

**Каждое действие в UI должно иметь трассировку до мандата.**

- **Область:** сценарии UI/UX (scenarios.md) и описание ролей.
- **Условие:** для каждой кнопки/действия в UI должно быть указано: какой мандат требуется, какой переход выполняется, какой метод вызывается.
- **Проверка:** при генерации документации - проверка наличия цепочки трассировки.

### R-DOC-GEN-ROLE-STRUCTURE

**Роли должны описываться в структурированном формате.**

- **Область:** файлы ролей (analyst.md, reviewer.md, ai_developer.md).
- **Условие:** описание роли должно содержать разделы: Purpose, Domain Visibility, Capabilities, Intent & KPI, Collaboration.
- **Проверка:** при генерации документации - проверка наличия всех обязательных разделов.

### R-DOC-GEN-GLOSSARY-INHERITANCE

**Глоссарий должен содержать наследование от [Base] Document и [Base] Participant.**

- **Область:** glossary/core/README.md и glossary/domain/README.md.
- **Условие:** в Core Glossary должны быть определены [Base] Document и [Base] Participant с атрибутами.
- **Проверка:** при генерации документации - проверка наличия базовых терминов.

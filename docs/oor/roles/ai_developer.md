# Роль: AI-Developer (ИИ-разработчик)

**Идентификатор роли:** `AI-Developer`
**Abstract:** Нет - роль назначается напрямую (но является автоматизированной, не человеческой).

## 1. Purpose (Назначение)

**Исполнитель (автоматизированная роль).** AI-Developer - это не человек, а ИИ-агент (например, в среде Cursor). Роль получает контекст из OOR-IDE (требования, домены, правила) и выполняет генерацию кода, рефакторинг и поддержку трассируемости кода к требованиям.

## 2. Domain Visibility (Видимость в домене)

### Domain_Module
- **Read:** [id, project_id, name, description, status, method_definition_ids, created_at, updated_at] (только при `status === Validated` или `status === Frozen`)
- **Write:** []

### Requirement_Project
- **Read:** [id, title, description, created_at, updated_at, domain_module_ids] (только если связанные Domain_Module имеют статус Validated/Frozen)
- **Write:** []

### Method_Definition
- **Read:** [id, domain_module_id, name, description, rule_schema] (только в рамках Validated/Frozen Domain_Module)
- **Write:** []

### Authorization_Mandate
- **Read:** [id, code, role_id, transition_id, method_definition_id, scope] (для понимания прав доступа)
- **Write:** []

## 3. Capabilities (Возможности)

### 3.1 ReadValidatedRequirements()
Чтение утверждённых требований и артефактов.
- **Мандаты:** M-OOR-VIEW-VALIDATED
- **Описание:** Доступ на чтение Domain_Module, Requirement_Project и Method_Definition при `status === Validated` или `status === Frozen`.

### 3.2 GenerateCodeFromSpecifications()
Генерация кода на основе OOR-спецификаций.
- **Мандаты:** M-OOR-VIEW-VALIDATED
- **Описание:** Использование контекста из Validated/Frozen артефактов для генерации кода, соответствующего требованиям.

### 3.3 RefactorCode()
Рефакторинг существующего кода с сохранением трассируемости.
- **Мандаты:** M-OOR-VIEW-VALIDATED
- **Описание:** Модификация существующего кода с обновлением ссылок на Method_Definition и сохранением соответствия требованиям.

### 3.4 SyncWithOORContext()
Использование панели AI-Sync для получения актуального контекста.
- **Мандаты:** M-OOR-VIEW-VALIDATED
- **Описание:** Доступ к панели AI-Sync для формирования контекста из Validated/Frozen артефактов.

### 3.5 ValidateCodeAgainstRules()
Проверка сгенерированного кода на соответствие правилам домена.
- **Мандаты:** M-OOR-VIEW-VALIDATED
- **Описание:** Валидация кода против правил, определённых в Rules.md домена.

### 3.6 MaintainTraceability()
Поддержка трассируемости кода к требованиям (комментарии, ссылки).
- **Мандаты:** M-OOR-VIEW-VALIDATED
- **Описание:** Добавление и поддержка ссылок на Method_Definition в сгенерированном коде.

## 4. Intent & KPI (Цель и метрики)

**Фокус:** Качество и соответствие кода требованиям.

**KPI (Key Performance Indicators):**
- Процент покрытия требований сгенерированным кодом
- Количество нарушений инвариантов в сгенерированном коде
- Время генерации кода для нового доменного модуля
- Качество трассируемости (процент методов кода с явными ссылками на Method_Definition)

## 5. Collaboration (Взаимодействие)

- **С Reviewer:** Получает от Reviewer (через SSOT) актуальную версию требований для исполнения. Результаты работы (код) могут быть предметом проверки Reviewer на соответствие требованиям.
- **С Analyst:** Косвенное взаимодействие через требования. AI-Developer использует артефакты, созданные Analyst, но не взаимодействует напрямую.
- **С другими AI-Developer:** Параллельная работа над разными частями системы, совместное использование контекста.

## 6. Ограничения

- Не создаёт и не изменяет требования, глоссарий или мандаты (это роли Analyst и Reviewer).
- Не переводит артефакты в статус Validated или Frozen (это роли Analyst/Reviewer).
- Не имеет доступа к артефактам в статусе Draft.
- Действия ограничены чтением и кодом; все изменения требований проходят через человека (Analyst/Reviewer).
- Не может нарушать инварианты, описанные в Rules.md.

## 7. Рабочий процесс (AI-Sync)

AI-Developer использует панель **AI-Sync** в OOR-IDE для получения контекста:

1. **Выбор артефакта:** AI-Developer выбирает Domain_Module в статусе Validated/Frozen
2. **Формирование контекста:** Система автоматически формирует контекст, включающий:
   - Глоссарий (Core + Domain)
   - Сущности выбранного артефакта
   - Переходы и правила
   - Мандаты и роли
3. **Копирование в IDE:** Контекст копируется в буфер обмена и вставляется в чат Cursor/Windsurf
4. **Генерация кода:** ИИ-агент использует контекст для генерации кода, соответствующего требованиям

## 8. Примеры использования

1. **Генерация кода для нового доменного модуля:**
   - Reviewer утверждает Domain_Module "OrderManagement" (статус Validated)
   - AI-Developer через AI-Sync получает контекст: сущности Order, OrderItem; переходы createOrder, cancelOrder; правила валидации
   - На основе контекста генерирует код: классы Order, OrderItem, методы createOrder(), cancelOrder() с соблюдением инвариантов

2. **Рефакторинг с сохранением трассируемости:**
   - При изменении требования (после утверждения новой версии Reviewer)
   - AI-Developer получает обновлённый контекст через AI-Sync
   - Выполняет рефакторинг кода, обновляя комментарии и ссылки на Method_Definition

3. **Проверка соответствия кода требованиям:**
   - AI-Developer анализирует существующий код на предмет соответствия текущим Validated требованиям
   - Формирует отчёт о расхождениях для Reviewer

## 9. Техническая интеграция

- **Интерфейс:** Панель AI-Sync в OOR-IDE
- **Формат контекста:** Markdown с структурированными разделами
- **Интеграция с IDE:** Через буфер обмена или API Cursor/Windsurf
- **Конфигурация:** Файл `.cursorrules` в корне проекта, ссылающийся на структуру `docs/oor/`

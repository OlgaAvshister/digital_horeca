# Роль: Reviewer (Рецензент)

**Идентификатор роли:** `Reviewer`
**Abstract:** Нет - роль назначается напрямую.

## 1. Purpose (Назначение)

**Контролёр качества и SSOT (Single Source of Truth).** Reviewer проверяет артефакты, созданные Analyst, на полноту, непротиворечивость и соответствие методологии OOR. После проверки утверждает переход артефактов в статус **Validated** и при необходимости **Frozen**, обеспечивая единый источник истины для AI-Developer и других потребителей требований.

## 2. Domain Visibility (Видимость в домене)

### Domain_Module
- **Read:** [id, project_id, name, description, status, method_definition_ids, created_at, updated_at] (все статусы: Draft, Validated, Frozen)
- **Write:** [status] (только через переходы состояний)

### Requirement_Project
- **Read:** [id, title, description, created_at, updated_at, domain_module_ids]
- **Write:** []

### Method_Definition
- **Read:** [id, domain_module_id, name, description, rule_schema]
- **Write:** []

### Authorization_Mandate
- **Read:** [id, code, role_id, transition_id, method_definition_id, scope] (для понимания прав доступа)
- **Write:** []

## 3. Capabilities (Возможности)

### 3.1 ReviewDomainModule()
Проверка доменного модуля на соответствие критериям качества.
- **Мандаты:** M-OOR-VIEW-DRAFT, M-OOR-VIEW-VALIDATED
- **Описание:** Анализ Domain_Module на полноту, непротиворечивость и соответствие методологии OOR.

### 3.2 ApproveDomainModule()
Утверждение артефакта (перевод Draft → Validated).
- **Мандаты:** M-OOR-ACT-APPROVE
- **Описание:** Инициация перехода состояния Domain_Module из Draft в Validated после успешной проверки.

### 3.3 RejectDomainModule()
Возврат артефакта на доработку (перевод Validated → Draft).
- **Мандаты:** M-OOR-ACT-REJECT
- **Описание:** Инициация перехода состояния Domain_Module из Validated в Draft при обнаружении недостатков.

### 3.4 FreezeDomainModule()
Фиксация версии (перевод Validated → Frozen).
- **Мандаты:** M-OOR-ACT-FREEZE
- **Описание:** Инициация перехода состояния Domain_Module из Validated в Frozen для создания неизменяемой версии.

### 3.5 ViewAllDrafts()
Чтение всех черновиков (включая созданные другими Analyst).
- **Мандаты:** M-OOR-VIEW-DRAFT
- **Описание:** Доступ на чтение Domain_Module и связанных сущностей при `status === Draft`, независимо от создателя.

### 3.6 ViewValidatedArtifacts()
Чтение утверждённых артефактов.
- **Мандаты:** M-OOR-VIEW-VALIDATED
- **Описание:** Доступ на чтение Domain_Module и связанных сущностей при `status === Validated` или `status === Frozen`.

## 4. Intent & KPI (Цель и метрики)

**Фокус:** Качество требований и поддержание SSOT.

**KPI (Key Performance Indicators):**
- Среднее время проверки артефакта (от получения до утверждения/возврата)
- Процент артефактов, возвращённых на доработку (качество первичных требований)
- Количество обнаруженных противоречий в требованиях
- Время реакции на запросы Analyst

## 5. Collaboration (Взаимодействие)

- **С Analyst:** Проверяет и утверждает результаты работы Analyst. Предоставляет обратную связь по качеству требований. Может инициировать возврат на доработку.
- **С AI-Developer:** Предоставляет доступ только к утверждённым (Validated/Frozen) артефактам как к единственному источнику истины. Гарантирует, что AI-Developer работает с корректными требованиями.
- **С другими Reviewer:** Совместная проверка сложных артефактов, консилиумы по спорным требованиям.

## 6. Ограничения

- Не создаёт первичный контент требований (это Analyst); Reviewer только проверяет и утверждает.
- Не может редактировать содержимое артефактов (только изменять статус через переходы).
- Действия ограничены мандатами (см. `domains/oor_manager/Mandates.md`).
- Не может переводить артефакты из Frozen обратно в Validated или Draft (только через создание новой версии Analyst).

## 7. Критерии приёмки (Checklist)

Перед утверждением артефакта (перевод Draft → Validated) Reviewer проверяет:

1. **Полнота глоссария:** Все термины из артефакта определены в Core или Domain Glossary.
2. **Непротиворечивость:** Нет конфликтующих правил в Rules.md.
3. **Трассируемость:** Каждое действие имеет привязку к мандату, каждый мандат ссылается на существующий переход.
4. **Соблюдение методологии:** Структура документации соответствует стандартам OOR (наследование от [Base] Document, правильные префиксы мандатов).
5. **Техническая корректность:** UUID ссылки валидны, JSON схемы корректны.

## 8. Примеры использования

1. **Проверка и утверждение артефакта:**
   - Analyst отправляет Domain_Module "OrderManagement" в статусе Draft на проверку
   - Reviewer проверяет артефакт по критериям приёмки
   - При соответствии критериям Reviewer выполняет переход Draft → Validated (мандат M-OOR-ACT-APPROVE)
   - Артефакт становится SSOT и доступен AI-Developer

2. **Возврат на доработку:**
   - При обнаружении недостатков Reviewer выполняет переход Validated → Draft (мандат M-OOR-ACT-REJECT)
   - Добавляет комментарии по необходимым исправлениям
   - Артефакт возвращается Analyst для доработки

3. **Фиксация версии:**
   - После утверждения артефакта и необходимости его неизменяемости Reviewer выполняет переход Validated → Frozen (мандат M-OOR-ACT-FREEZE)
   - Замороженная версия используется для релиза или как эталон для аудита
